# 問題分析・課題整理フレームワーク

## 概要
問題を構造化して分析し、課題を明確にするための思考フレームワーク。

## 思考の進め方

### 1. 問題の明確化
- 何が問題なのかを具体的に定義する
- 事実と意見を区別する
- 問題が誰にとっての問題なのかを明確にする

### 2. 現状把握
- 現在の状況を客観的に把握する
- 定量的・定性的なデータを収集する
- 関係者の視点を理解する

### 3. 理想状態の定義
- あるべき姿、理想の状態を明確にする
- 現状とのギャップを特定する
- 目標を具体的に設定する

### 4. 根本原因の特定
- 表面的な症状と根本原因を区別する
- 因果関係を分析する
- 複数の要因を考慮する

### 5. 影響範囲の評価
- 誰が影響を受けるか
- どの程度の影響があるか
- 時間的な広がりは

### 6. 課題の優先順位付け
- 緊急性と重要性で評価する
- 実現可能性を考慮する
- リソースとのバランスを取る

## 確認すべき観点

### 問題の性質
- 技術的問題か、人的問題か、プロセスの問題か
- 構造的な問題か、一時的な問題か
- 予測可能だったか、予測不能だったか

### ステークホルダー
- 誰が影響を受けるか（顧客、チーム、経営層など）
- 誰が問題解決に関与すべきか
- それぞれの関心事は何か

### 制約条件
- 時間的制約
- 予算・リソースの制約
- 技術的制約
- 組織的・政治的制約

### リスク
- 放置した場合の影響
- 問題が悪化する可能性
- 他の領域への波及リスク

## 使える思考ツール

### 5W1H
- **Why**: なぜこの問題が発生したのか？
- **What**: 何が問題なのか？
- **Where**: どこで問題が発生しているのか？
- **When**: いつ問題が発生したのか？いつまでに解決すべきか？
- **Who**: 誰が関係しているのか？誰が影響を受けるのか？
- **How**: どのように問題が発生したのか？

### なぜなぜ分析（5 Whys）
問題に対して「なぜ？」を5回繰り返し、根本原因に到達する。

例:
1. なぜシステムが停止したのか？ → サーバーがダウンしたから
2. なぜサーバーがダウンしたのか？ → メモリ不足が発生したから
3. なぜメモリ不足が発生したのか？ → メモリリークがあったから
4. なぜメモリリークがあったのか？ → コードにバグがあったから
5. なぜバグがあったのか？ → レビュープロセスが不十分だったから

### 特性要因図（フィッシュボーン図）
問題の要因を以下のカテゴリで整理:
- 人（Man）
- 機械（Machine）
- 方法（Method）
- 材料（Material）
- 測定（Measurement）
- 環境（Environment）

### MECE（Mutually Exclusive, Collectively Exhaustive）
- 問題を漏れなく、ダブりなく分解する
- 構造化された分析を実現する

### マトリクス分析
2軸で問題を分類・評価:
- 緊急度 × 重要度
- 影響度 × 実現可能性
- コスト × 効果

## 多様な視点

### 短期 vs 長期
- 短期的に見た場合の影響
- 長期的に見た場合の影響
- 一時的な対症療法か、恒久的な解決か

### 内部 vs 外部
- 組織内部の問題か
- 外部環境の変化による問題か
- 両方の要因が絡んでいるか

### 楽観的 vs 悲観的
- 最良のシナリオ
- 最悪のシナリオ
- 現実的なシナリオ

### 定量的 vs 定性的
- 数値で測定できる影響
- 数値化しにくい影響（モチベーション、評判など）
- 両方をバランスよく考慮する

## 憲法の原則の適用

### 誠実性
- 事実と推測を明確に区別する
- 不都合な真実も直視する
- 自分のバイアスを認識する

### 多様性
- 複数のステークホルダーの視点を取り入れる
- 異なる専門性からの意見を求める
- 自分とは異なる解釈も検討する

### 構造化
- 問題を論理的に分解する
- 因果関係を明確にする
- 分析プロセスを可視化する

### 実践性
- 実行可能な解決策につながる分析
- リソースと制約を現実的に評価
- 理論だけでなく実践を見据える

### 継続的改善
- 過去の類似問題から学ぶ
- 分析手法自体を改善する
- フィードバックループを組み込む

## チェックリスト

- [ ] 問題を具体的に定義できたか
- [ ] 事実と意見を区別したか
- [ ] 根本原因を特定したか（表面的な症状だけで終わっていないか）
- [ ] 複数のステークホルダーの視点を考慮したか
- [ ] 定量的・定性的な影響を評価したか
- [ ] 時間軸（短期・長期）を考慮したか
- [ ] 制約条件を明確にしたか
- [ ] 解決可能な課題として整理できたか
- [ ] 優先順位を付けたか
- [ ] 次のアクションが明確か

## ペルソナとの関連

### 事業責任者の視点
- ビジネスへの影響度を定量化する
- 競合や市場環境との関連を分析する
- ROIの観点から優先順位を評価する

### チームメンバーの視点
- 技術的な実現可能性を評価する
- 実装・運用の複雑性を考慮する
- チームのキャパシティとスキルを確認する

### エンドユーザーの視点
- ユーザー体験への影響を分析する
- ユーザーの痛み・困りごとを理解する
- 使いやすさや価値提供の観点で評価する

## 次のステップ

問題分析が完了したら:
1. 課題を明確にドキュメント化する
2. 関係者と認識を共有する
3. 解決策の検討（ブレストや意思決定）に進む
4. 定期的に状況を再評価する
